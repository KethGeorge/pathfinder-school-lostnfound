## 1. 项目概述

### 1.1 项目背景

```
校园面积大、建筑物多，新生和访客经常找不到目的地。
同时校园内失物招领信息分散，缺乏统一平台。
本系统旨在提供：
├── 校内智能寻路导航
├── 失物招领信息发布与查询
└── 统一的用户管理与后台管理
```

### 1.2 项目目标

```
├── 为师生提供校内地点搜索与路径导航功能
├── 提供失物招领信息发布、搜索、认领的完整流程
├── 提供管理员后台进行数据管理与统计
└── 系统采用 B/S 架构，通过浏览器即可使用
```

### 1.3 术语定义

| 术语   | 定义                               |
| ---- | -------------------------------- |
| MVP  | Minimum Viable Product，最小可行产品    |
| JWT  | JSON Web Token，用户身份认证令牌          |
| ORM  | Object-Relational Mapping，对象关系映射 |
| CRUD | Create/Read/Update/Delete，增删改查   |
| DTO  | Data Transfer Object，数据传输对象      |
| VO   | View Object，视图对象（返回给前端的数据结构）     |

## 4. 数据库设计

### 4.0 设计约定

```
关联方式：统一采用【逻辑关联】，不使用物理外键约束
├── 表间关系（如 lost_found.user_id → user.id）仅在程序层维护
├── ER 图中的 FK 标注表示"逻辑外键"，非数据库物理约束
├── 好处：建表无先后依赖、便于清库/重导数据、符合主流规范
└── 代价：数据完整性由 Service 层保证（插入前校验引用是否存在）

索引策略：按"是否作为查询条件"决定，与关联方式无关
├── 高频 WHERE/JOIN 字段建索引（如 map_edge.from_node/to_node）
└── 仅查出后使用、不作过滤条件的字段不建索引（如 nearest_node_id）
```

### 4.2 建表 SQL

```sql
-- ============================================
-- 校园寻路与失物招领系统 数据库初始化脚本
-- 数据库：campus
-- 字符集：utf8mb4
-- ============================================

CREATE DATABASE IF NOT EXISTS campus
  DEFAULT CHARACTER SET utf8mb4
  DEFAULT COLLATE utf8mb4_unicode_ci;

USE campus;


-- -------------------------------------------
-- 1. 用户表
-- -------------------------------------------
CREATE TABLE `user` (
    `id`          BIGINT       NOT NULL AUTO_INCREMENT  COMMENT '用户ID',
```

### 4.3 字段设计说明

```
物品分类枚举 (category)：
├── card      - 校园卡/证件
├── umbrella  - 雨伞
├── phone     - 手机/电子设备
├── keys      - 钥匙
├── bag       - 包/书包
├── book      - 书籍/文具
├── clothing  - 衣物
└── other     - 其他

失物状态枚举 (status)：
├── OPEN     - 进行中（默认）
├── CLAIMED  - 已认领
└── CLOSED   - 已关闭

用户角色枚举 (role)：
├── USER     - 普通用户（默认）
└── ADMIN    - 管理员

节点类型枚举 (map_node.type)：
├── BUILDING  - 教学楼/办公楼
├── CANTEEN   - 食堂
├── DORM      - 宿舍
├── GATE      - 校门
├── CROSSROAD - 路口（不显示给用户，仅用于寻路计算）
└── OTHER     - 其他（操场、花园等）

边类型枚举 (map_edge.edge_type)：
├── ROAD      - 车行道（骑行+步行都可走）
├── PATH      - 人行道/捷径（仅步行）
├── INDOOR    - 室内通道
├── STAIRS    - 楼梯
└── ELEVATOR  - 电梯

可通行方式枚举 (map_edge.accessible_by)：
├── WALK      - 仅步行
├── BIKE      - 仅骑行（一般不用）
└── BOTH      - 步行和骑行都可

节点索引约定（前后端统一）：
├── 正整数：表示路口节点（crossingData数组索引）
└── 负整数：表示建筑节点（-idx-1 对应 buildingData数组索引）
    示例：-1 表示 buildingData[0]（第一个建筑）
          0 表示 crossingData[0]（第一个路口）
```

## 5. 接口设计

### 5.1 通用约定

#### 5.1.1 基础URL

```
开发环境：http://localhost:8080/api
生产环境：http://你的域名或IP/api
```

#### 5.1.2 认证方式

```
除登录和注册外，所有接口需携带 Token：
Header: Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...

Token 有效期：24小时
Token 过期返回 401 状态码
```

#### 5.1.3 统一响应格式

```json
// 成功
{
    "code": 200,
    "message": "success",
    "data": { ... }
}

// 失败
{
    "code": 400,
    "message": "具体错误信息",
    "data": null
}
```

#### 5.1.4 统一分页格式

```json
{
    "code": 200,
    "data": {
        "list": [ ... ],
        "total": 100,
        "page": 1,
        "pageSize": 10
    }
}
```

#### 5.1.5 错误码定义

| 错误码 | 含义        | 场景                      |
| --- | --------- | ----------------------- |
| 200 | 成功        | 所有成功请求                  |
| 400 | 参数错误/业务错误 | 参数校验失败、业务规则不满足          |
| 401 | 未认证       | Token缺失、Token过期、Token无效 |
| 403 | 无权限       | 普通用户访问管理接口、操作他人数据       |
| 404 | 资源不存在     | ID不存在、路径不可达             |
| 500 | 服务器内部错误   | 未捕获的异常                  |

#### 5.1.6 时间格式

```
统一使用 ISO 8601 格式：
"2024-12-20T15:00:00"
```

## 8. 寻路算法设计

### 8.1 算法选择

```
选择 Dijkstra 算法（简化实现）

理由：
├── 校园规模小（节点 < 200，边 < 500）
├── Dijkstra 可以保证最短路径
├── 实现简单，容易理解和调试
├── 简化版时间复杂度 O(V²)，在该规模下仍可接受（< 10ms）
├── 使用数组模拟优先队列（代码简单，性能够用）
└── 不需要 A* 的启发式优化（图太小了）

注：如需优化可使用堆实现真正的优先队列（O((V+E)logV)）
```

### 8.2 导航模式

```
步行模式 (walk)：
├── 可走所有边（ROAD + PATH）
├── 优先选择PATH类型（人行道捷径）
└── 速度：1.2 m/s

骑行模式 (bike)：
├── 仅可走ROAD类型边
├── 不能走PATH类型边（人行道/楼梯）
└── 速度：3.0 m/s

边过滤逻辑：
if mode == 'bike':
    edges = edges.filter(e => e.accessible_by in ['BIKE', 'BOTH'])
else:  // walk
    edges = edges.filter(e => e.accessible_by in ['WALK', 'BOTH'])
```

### 8.3 数据结构

```
图的表示：邻接表

Map<Long, List<Edge>> graph

其中 Edge = {
    toNodeId: Long,        // 目标节点
    distance: Double,      // 边权重（像素距离）
    edgeType: String,      // ROAD/PATH
    accessibleBy: String   // WALK/BIKE/BOTH
}

示例：
graph = {
    1(图书馆):  [(11,路口A, 120px, ROAD, BOTH)],
    11(路口A):  [(1,图书馆, 120px, ROAD, BOTH), 
                (4,教学楼A, 80px, PATH, WALK),  // 人行道捷径
                (5,教学楼B, 60px, ROAD, BOTH), 
                (12,路口B, 150px, ROAD, BOTH)],
    12(路口B):  [(11,路口A, 150px, ROAD, BOTH),
```

### 8.4 算法伪代码

```
function dijkstra(graph, startId, endId):
    dist = {}          // 起点到各节点的最短距离
    prev = {}          // 各节点的前驱节点（用于回溯路径）
    visited = Set()    // 已确定最短路径的节点
    pq = PriorityQueue()  // 优先队列，按距离排序

    dist[startId] = 0
    pq.add((0, startId))

    while pq is not empty:
        (d, u) = pq.poll()     // 取距离最小的节点

        if u == endId:
            break              // 找到终点，提前结束

        if u in visited:
            continue
        visited.add(u)

        for each edge (u → v, weight) in graph[u]:
            newDist = dist[u] + weight
            if newDist < dist.getOrDefault(v, INF):
                dist[v] = newDist
                prev[v] = u
                pq.add((newDist, v))

    // 回溯路径
    path = []
    node = endId
    while node != startId:
        path.addFirst(node)
        node = prev[node]
    path.addFirst(startId)

    return {
        path: path,
        distance: dist[endId]
    }
```

### 8.5 距离和时间计算

```
坐标系统：像素坐标（非地理坐标）

像素距离转实际距离：
├── 需要设定转换系数 PIXEL_TO_METER
├── 测量地图上已知距离建立比例尺
├── 示例：假设地图宽度2000px对应实际1000m
│   则 PIXEL_TO_METER = 1000 / 2000 = 0.5
└── realDistance(米) = pixelDistance * PIXEL_TO_METER

步行速度：1.2 m/s（约 4.3 km/h）
骑行速度：3.0 m/s（约 10.8 km/h）

时间计算：
walkDuration(秒) = realDistance(米) / 1.2
bikeDuration(秒) = realDistance(米) / 3.0
durationText = Math.ceil(duration / 60) + "分钟"

示例（假设PIXEL_TO_METER = 0.5）：
456像素 → 228米 → 190秒 → "约3分钟"（步行）
456像素 → 228米 → 76秒 → "约2分钟"（骑行）

注意：
├── 前端显示时可直接使用像素距离（相对距离）
├── 或通过转换系数计算实际距离（绝对距离）
└── 建议在系统配置中存储 PIXEL_TO_METER 参数
```

### 8.6 简化实现说明

```
当前HTML实现使用简化的优先队列：

优点：代码简单，易于理解

优化建议（可选）：
```


#### 8.7.2 就近吸附算法（复用 getClosetPoint）

```javascript
// 当前 HTML connectHelper.getClosetPoint() 已实现
// 后端等价逻辑：遍历所有节点，返回距离(x,y)最近的节点ID
function findNearestNode(x, y) {
    let minDist = Infinity, nearestId = null;
    for (const node of allNodes) {          // crossing + building
        const d = (node.x - x)² + (node.y - y)²;  // 距离平方即可比较
        if (d < minDist) { minDist = d; nearestId = node.id; }
    }
    return nearestId;
}
```

```
  → 查询用户信息放入上下文
```

## 13. Git 规范

### 13.1 分支策略

```
main（保护分支）
  │
  └── develop（开发主线）
       │
       ├── feature/user-auth          A+D 负责
       ├── feature/lost-found         B+E 负责
       ├── feature/navigation         C+F 负责
       ├── feature/admin              C+E 负责
       └── hotfix/xxx                 紧急修复
```

### 13.2 Commit 规范

```
格式：<type>(<scope>): <subject>

type:
├── feat      新功能
├── fix       修Bug
├── docs      文档
├── style     格式（不影响逻辑）
├── refactor  重构
├── test      测试
└── chore     杂项（构建、依赖等）

scope（可选）:
├── auth        认证模块
├── user        用户模块
├── lost-found  失物招领模块
├── navigation  导航模块
├── admin       管理模块
└── common      公共模块

示例：
feat(lost-found): 添加失物招领列表页分页功能
fix(auth): 修复Token过期后未跳转登录页的问题
docs(readme): 更新项目部署文档
```

### 13.3 PR 规范

```
PR 模板：
├── 标题：feat(模块): 一句话概括
├── 改动说明：列出具体改动点
├── 截图：页面变更必须贴图
├── 测试情况：自测了哪些场景
└── 注意事项：需要 reviewer 关注的点

合并规则：
├── 至少 1 人 Approve 才能合并
├── 自己不能 Approve 自己的 PR
├── 合并后删除 feature 分支
└── 有冲突必须解决后才能合并
```

### 13.4 GitHub 保护规则

```
对 main 分支：
├── ☑ Require pull request before merging
├── ☑ Require approvals: 1
├── ☑ Do not allow bypassing the above settings
└── ☑ 禁止 force push

对 develop 分支（建议也保护）：
├── ☑ Require pull request before merging
└── ☑ Require approvals: 1
```

```
│                                            │
└────────────────────────────────────────────┘
```

#### 部署步骤

```bash
# 1. 本地打包后端
cd campus-backend
mvn clean package -DskipTests
# 产物：target/campus-system-0.0.1-SNAPSHOT.jar

# 2. 本地打包前端
cd campus-frontend
npm run build
# 产物：dist/ 目录

# 3. 上传到服务器
scp target/campus-system-0.0.1-SNAPSHOT.jar root@服务器IP:/opt/app/
scp -r dist/* root@服务器IP:/var/www/campus/

# 4. 服务器上启动后端
cd /opt/app
nohup java -jar campus-system-0.0.1-SNAPSHOT.jar \
  --spring.profiles.active=prod > app.log 2>&1 &

# 5. 配置 Nginx（见下方配置）
sudo nginx -t
sudo systemctl reload nginx
```

#### Nginx 配置

```nginx
server {
    listen 80;
    server_name xxx.xxx.xxx.xxx;

    # 前端静态文件
    location / {
        root /var/www/campus;
        index index.html;
        try_files $uri $uri/ /index.html;
    }

    # 后端API代理
    location /api/ {
        proxy_pass http://localhost:8080/api/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # 文件上传大小限制
        client_max_body_size 10m;
    }

    # 上传的图片文件
    location /uploads/ {
        alias /opt/app/uploads/;
        expires 30d;
    }

    # Knife4j 接口文档（可选，生产环境可关闭）
    location /doc.html {
        proxy_pass http://localhost:8080/doc.html;
    }
}
```
